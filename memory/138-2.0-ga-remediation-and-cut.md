# 138 - 2.0 GA remediation and cut

Date: 2026-06-17

Closes the 2.0 line: a pre-release review (full review + stability run before
cutting GA) returned NO-GO on one blocker plus a set of confirmed non-blocking
fixes. Owner decision: "Blocker + all confirmed fixes, then cut GA." All were
remediated, the suite was stabilized, and the 20 component packages were flipped
from `2.0.0-preview.1` to `2.0.0` (final). The engine stays `1.3.0` (additive).

## Blocker

- **State clock missed by the TimeProvider sweep.** `FluxFlow.Components.State`
  still shipped a bespoke `IStateClock`/`SystemStateClock`; an earlier commit had
  falsely claimed it was migrated. Migrated it to `System.TimeProvider`
  (`UseClock(TimeProvider)`; `StateReducerNode` uses `_clock.GetUtcNow()`),
  deleted the interface + implementation + `RecordingStateClock`, and rewrote the
  clock test against `FakeTimeProvider`.

## Confirmed fixes

- **Connection-node dispose-race lease leak** (MQTT/HTTP/Storage): a freshly
  established handle could leak if the node was disposed/disconnected, or the
  establish was superseded, while the open was in flight. Added a
  decide-and-publish guard (publish only if not disposed/not user-disconnected/
  not superseded/handle-null; otherwise drop and dispose the fresh handle once)
  and tolerate the single-flight gate being disposed mid-establish
  (`ObjectDisposedException` still disposes the lease). Added failure-path tests
  to each: connect-fault -> Faulted -> retry, disconnect-wins-during-establish,
  dispose-racing-in-flight-connect.
- **No-bespoke-clock release guard**: `FluxFlow.Release.Tests/
  BespokeClockInterfaceTests` scans `src/` for any `interface IXxxClock` and
  asserts the set is empty. This is the regression guard that would have caught
  the State miss.
- **Mapper diagnostic**: restored the descriptive `MapperFailed` message for
  wrong-type/null compiled-expression results (names the expected output type)
  in `FlowMapperNode`; engine unchanged.
- **Stale packaged READMEs** (Mqtt/Http/Storage/Timers): rewritten to the 2.0
  shapes — separate connection resources, required connection reference, host-
  driven `ConnectAsync`/`DisconnectAsync` (no auto-connect), `UseClock(
  TimeProvider)` — and removed the 1.x clock interfaces and inline-connection
  options.

## Test stability (three flakes fixed; root causes, not timeout band-aids)

Surfaced only under heavy parallel solution-wide load:

- **FlowWindow real-clock duration**: `Window_EmitsWhenTimeElapsedWithoutNextInput`
  asserted `Duration >= 20ms` for a 25ms window, but `GetUtcNow()` snaps to the
  ~15.6ms Windows system tick, so a single-tick duration (~15.6ms) trips it.
  Assert only positive elapsed; the exact 25ms duration is pinned deterministically
  with a fake clock in `RoutingClockTests`.
- **FlowJoin one-shot clock fault**: `Join_ReportsProcessingFailureAndContinues`
  used a `ThrowingTimeProvider` that throws on the first `GetUtcNow()` globally,
  but the join drains Left/Right concurrently so the throw landed on a
  non-deterministic message, sometimes breaking the pair that should match. Fixed
  by sending the failing message alone and awaiting its error first (deterministic
  one-shot consumption), then sending the matching pair.
- **StorageStore secondary fanout ports**: `Operations_RoundTripThroughTheOpenedStore`
  (and the missing-factory diagnostic test) read `Found`/`Records`/`Diagnostics`
  with `TryReceive` (no wait); those ports are independent fanout sources pumped
  on their own tasks, so `Node.Completion`/connect-return does not guarantee
  delivery. Await the item instead. (Note: `OutputPort_...` in the engine test
  awaits the port's own pump `Completion` before `TryReceive`, so it is correct.)
- Standardized positive async waits to 30s in Sessions/Sources/Timers/Expectations
  (matching Routing), and relaxed the Expectations real-time poll bound to 30000ms.

Stability after fixes: Routing 50/50, Storage 40/40, full solution 15/15
consecutive green at 717 tests (Release).

## GA cut

- Flipped `2.0.0-preview.1` -> `2.0.0` in 20 component csprojs and their 20
  CHANGELOG headings (`get-release-notes`/preflight key off the heading, so they
  must match the csproj version). No inter-package version pins (ProjectReference
  throughout); historical memory records keep their `preview.1` text.
- Engine ships `1.3.0` (additive: `GetResource<T>`, `IFlowExpressionEngine.
  Compile<T>` default interface, error/event broadcast fanout). Components ship
  `2.0.0`.
- Publish order: engine `1.3.0` first (components' ProjectReference bakes a
  `>= 1.3.0` floor into the nupkg dependency), then the 20 `2.0.0` components;
  verify the feed (flat-container index pre-check, see [[134-feed-verify-index-precheck]]).
