# Wave 2 (2.0) Plan — Expression-Node Compile-Once + Factory Relocation

Date: 2026-06-15

Review-ready scope for the breaking 2.0 work (principles 1 & 2 from
`135-architecture-review-and-roadmap.md`). Produced by a per-node inspection
workflow + synthesis + skeptical critique (all claims spot-verified). No code
written yet — this is the plan to approve before implementation.

## Canonical transformation (every expression node)

Factory compiles the expression ONCE and injects a typed delegate; node holds
only the delegate.

- Before: node ctor takes `(options, IFlowExpressionEngine, IXxxContextFactory,
  XxxNodeContext)`; per message it builds a context and calls
  `engine.Evaluate(options.Expression!, ctx, type)` — re-parsing every message.
- After: factory builds `new ExpressionFlowPredicate<T>(expr, engine, adapter)`
  (or `ExpressionFlowMapper`, or a `engine.Compile<T>` closure) once and injects
  `IFlowPredicate<T>`/`IFlowMapper<,>`/`Func`. Node ctor drops the engine, the
  expression string, and the context factory.
- A one-line per-package adapter bridges the component's 2-arg
  `Create(input, nodeContext)` to the engine seam's 1-arg
  `IFlowMapContextFactory<TInput>.Create(input)` (the two shapes do not unify;
  adapter is required).

## 10 expression nodes

- Reuse `ExpressionFlowPredicate` (+adapter): `flow.filter`, `flow.when`
  (Control), `flow.assert` (Assertions).
- Reuse `ExpressionFlowMapper` (fed the node's own rich context, since
  `ExpressionFlowMapper.Map` ignores its input arg): `flow.mapper` (Mapping).
- Hand-wrapped `engine.Compile<T>` closures (richer context or route-key/object
  result): `state.reducer` (State, + a new `IFlowReducer`), `flow.counter`
  (Observability, conditional/nullable predicate), `flow.switch`,
  `flow.correlation` (conditional side selector), `flow.join` (two independent
  selectors) (Routing).

## JsonSchemaValidatorNode (principle 1 worst offender)

Move `File.ReadAllText` + `JsonSchema.FromText` from the node's `StartAsync`
into `JsonSchemaValidatorNodeFactory` (run once at build); inject a ready
`JsonSchema`. Drop `JsonSchemaValidatorContext.Options` so selectors stop seeing
raw config. Behavioral break: schema-load failure moves from `StartAsync` to
graph-build time.

## Co-located factory relocation (principle 1)

Move every `public static Create(RuntimeNodeFactoryContext…)` out of node files
into dedicated `*NodeFactory` classes: Http, Metrics, Storage(×4), Sessions(×3),
FileSystem(×4), Timers Interval+Schedule, Mqtt(×2), Payloads, Projections,
Expectations, State. Already correct (templates): Control, Assertions, Mapping,
Routing, Observability, Serialization, Sources, Validation, Timers
Debounce/Delay/Throttle. Note: "has a dedicated factory" ≠ "principle 2 done" —
Control/Assertions/Mapping factories still inject the raw engine (that's the
expression work, not relocation).

## Breaking-surface summary

- **JSON / options / registration: NOTHING breaks.** Options records, node-type
  ids, `Register*Components`, JSON shape, and ports are all unchanged. A
  JSON-only graph author sees zero source breaks (only diagnostic provenance and
  the validator's failure-timing change are observable).
- **Public node ctors change** (direct-construction path only): `FilterNode`,
  `WhenNode`, `FlowAssertionComponent` (both), `FlowMapperNode`, `FlowSwitchNode`,
  `FlowCorrelationNode`, `FlowJoinNode`.
- **Internal ctors** (no external impact): `StateReducerNode`, `FlowCounterNode`,
  `JsonSchemaValidatorNode`.
- **External contract retirements:** the `IXxxContextFactory` / `XxxNodeContext`
  2-arg shape and `JsonSchemaValidatorContext.Options` (+ selector `Select`
  shape) — affects external context-factory/selector authors.
- ~25 `public static Create` removed (in-repo only the modules call them).

Verdict: major-version in spirit (removed public ctors/contract members) with an
unusually gentle migration for graph authors.

## Sequencing (keep tests green at each step)

1. Pure relocations first (one package per PR; no behavior change).
2. `flow.counter` (internal ctors, template exists) — proves the closure shape.
3. `flow.filter` + `flow.when` together (share the Control adapter).
4. `flow.assert`. 5. `flow.mapper`. 6. `state.reducer` (heaviest).
7. Routing switch → correlation → join (join last — only node with direct-ctor
   tests to rewrite). 8. JsonSchemaValidator (parallel, independent).

Test doubles (`RecordingExpressionEngine`) implement only `Evaluate` and inherit
the default `Compile`, so compile-once preserves per-message interception —
almost no test-double rewrites; only two construction-site edits exist
(FlowJoinNodeTests).

## Important clarification on the "type-check / parse-once" payoff

The repo ships NO concrete `IFlowExpressionEngine` (host-provided). With the
default `Compile` (deferring to `Evaluate`), Wave 2's repo-side win is purely
the **decoupling** (nodes stop holding engine+string+context). The
**parse-once + build-time type-check** payoff is realized only when a consumer's
engine overrides `Compile<T>` to parse once and validate the result type. The
repo can demonstrate it via a reference/sample precompiling engine but cannot
realize it in shipped packages. The critique confirmed: a precompiling engine is
a payoff-enabler, NOT a blocking dependency for the migration.

## Version recommendation

One coordinated **2.0** wave; hold as unpublished `2.0.0-preview.*` until the
whole wave is green, then ship 2.0 once (shipping `public static Create` removals
under a 1.x minor would violate semver; State is both a relocation and an
expression node, so they must land together). The 1.x batch (Waves 0+1) stays
frozen/unpublished and can either ship first as 1.x or fold into the 2.0 cut —
owner decision.
