# Wave 3 (2.0) Plan — Connection Resource Components + TimeProvider

Date: 2026-06-15

Review-ready scope for the last 2.0 wave (principle 3 + the clock
consolidation). Produced by a scoping workflow + synthesis + a skeptical
critique that verified the load-bearing feasibility question. No code yet.

## Feasibility (verified)

The engine `$resources` mechanism supports the connection-resource pattern at
BUILD time: `ApplicationRuntimeBuilder` builds `definition.Resources` first and
passes the resolved resource nodes as `RuntimeNodeFactoryContext.Resources` to
every workflow factory, so an operation factory resolves its connection node
synchronously via `context.GetResource(name)`. Teardown disposes workflows
first, resources last — a shared client outlives its borrowers. Only engine
addition: an additive `GetResource<T>(NodeName)` typed downcast helper.

## A. Connection resource components

One resource node per backend in `$resources`: `mqtt.connection`,
`http.client`, `storage.store`. Each owns the connection-scoped state (client/
pool + reconnect + one health monitor); operation nodes (`mqtt.publish/
subscribe`, `http.request`, `storage.*`) carry only a name reference and
resolve via `GetResource<T>` at build. Each component defines its own handle
interface (`IMqttConnectionHandle`/`IHttpClientHandle`/`IStorageStoreHandle`);
the resource node implements it. The engine stays protocol-ignorant (no shared
resource-payload contract — `T` is the component's own handle).

### Ordering correction (critic finding — important)

The connection's client must be usable when an operation node starts. A
factory CANNOT set `Phase` — `Phase` is author JSON the builder forcibly
applies, so "connection starts before ops" is NOT automatic (default phase 0 =
same start group, enumeration order). Resolution: **lazy-connect through the
handle** — the handle exposes a connect-once accessor (e.g.
`ValueTask<IMqttClientAdapter> GetAdapterAsync(ct)`) so ordering stops
mattering (HTTP/Storage don't connect eagerly; MQTT subscribe awaits it in
StartAsync). Avoids an engine change, an authoring-phase requirement, and a
readiness-signal handshake. (Alternative considered and not preferred: a small
engine change to start all resources before workflow nodes.)

### Per-protocol worklist

- MQTT (template): new `MqttConnectionNode`/`MqttConnectionOptions`/
  `IMqttConnectionHandle`/`mqtt.connection` type + registration; move the
  client lease + the single health monitor + reconnect into it; `mqtt.publish/
  subscribe` drop `Connection`/`Reconnect`, keep required `connectionName`,
  share the adapter via `MqttClientLease.Shared` (no dispose).
- HTTP: new `http.client` owning one pooled `HttpClient`/`SocketsHttpHandler`
  + baseUrl/allowedHosts/redirects/timeout/pooling; `http.request` keeps
  per-request fields + a `client` ref. SECURITY: keep per-request
  `EnsureUrlAllowed` against the resource allow-list; op nodes may only narrow,
  never widen; auto-redirect stays disabled when an allow-list is set.
- Storage: new `storage.store` owning one adapter + its lifetime; `storage.*`
  ops keep a `store` ref. Fixes the FileSystem never-disposed cache and the
  SqlFile per-node duplication uniformly; retire `StorageStoreLease.Owned/
  Shared` and the process-wide `IStorageStoreFactory` registration.

Build-time validation: factories check the referenced name exists before
`GetResource<T>` and throw an actionable `FactoryFailed` error (the
`$resources` link validator doesn't see config-string refs).

## B. TimeProvider clock consolidation

Replace all 15 `IXxxClock` (+ `System*`/`Recording*` doubles) with
`System.TimeProvider` (in-box net8+; covers 100%: `GetUtcNow` +
`Task.Delay(delay, tp, ct)`; no `CreateTimer`). Clean 2.0 break (the `Clock`
getter type must change). Per `*ComponentOptions`: `UseClock(TimeProvider)`,
`Clock` retyped; node ctors retype the param; call sites `_clock.UtcNow` →
`GetUtcNow()`, `DelayAsync(d,ct)` → `Task.Delay(d,_clock,ct)`. Public context/
options records carrying a clock retype too (`HttpRequestSenderContext`,
`StorageStoreContext`, the two store-options). Tests: add
`Microsoft.Extensions.TimeProvider.Testing`, replace doubles with
`FakeTimeProvider` (note: queue-of-timestamps and delay doubles need behavioral
rewrites — `FakeTimeProvider` holds time until `Advance`). Genuine gap: the
`ThrowOnce*` fault-injection clocks need a tiny bespoke throwing `TimeProvider`.

## Breaking surface (all 2.0)

- Additive engine: `GetResource<T>()`.
- Wire format: MQTT op config loses `connection.*`/`reconnect.*` (hoist to an
  `mqtt.connection` resource + required `connectionName`); new `http.client`/
  `storage.store` types; HTTP connection options move to `http.client`.
- Removed: `Mqtt*Options.Connection/Reconnect`, `StorageStoreLease.Owned/
  Shared` + process-wide store registration, 15 `IXxxClock` + 15 `System*Clock`.
- Added: the three connection node types/options/handles.
- Behavioral: MQTT `ConnectionHealthChanged` now keyed to the connection node
  id (one health stream per connection — release-note it).
- `UseClock(IXxxClock)` → `UseClock(TimeProvider)`; clock-bearing context/
  options records retyped.

## Sequencing

1. `GetResource<T>` engine helper (additive, alone).
2. TimeProvider sweep (mechanical, many packages; Storage 3-csproj atomic
   commit; isolate the delay/fault test-double rewrites) — before connections
   so the new resource nodes are born on `TimeProvider`.
3. `mqtt.connection` (template, first `GetResource<T>` consumer, lazy-connect).
4. `http.client` (carries the redirect/allow-list security hazard).
5. `storage.store` (carries the FileSystem-leak/SqlFile-duplication fixes).

Reject the implicit-connection / implicit-store back-compat shims floated in
scoping — they add a parallel resolution path and undercut the clean 2.0 break;
document the JSON migration in the preview release notes instead.

Everything stays on the unpublished `2.0.0-preview.1` track.
